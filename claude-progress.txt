[2026-05-16] Serilog log<->trace correlation via custom enricher

  Bug: after we removed the DD .NET tracer (pure-OTel migration), the
  Serilog setup in both .NET services still had a stale comment
  claiming "DD_LOGS_INJECTION=true auto-injects dd.trace_id /

  services/agent-api-dotnet/Observability/DatadogTraceContextEnricher.cs  [NEW]
  services/mcp-server-dotnet/Observability/DatadogTraceContextEnricher.cs  [NEW]
    Identical 50-line ILogEventEnricher in both services (separate
    csproj files, duplication beats sharing for a class this small).
    Reads Activity.Current; writes top-level CLEF keys:
      dd.trace_id = lower 64 bits of W3C trace_id, base-10
      dd.span_id  = span_id, base-10
    Decimal-low-64 is the historic DD .NET tracer format; the agent's
    csharp log integration recognizes it verbatim. The high 64 bits
    of the 128-bit W3C trace_id are preserved in the OTel trace
    metadata and recombined at correlation time by DD's log pipeline.

  services/agent-api-dotnet/Observability/TelemetrySetup.cs
  services/mcp-server-dotnet/Observability/TelemetrySetup.cs

    comment also corrected on both sides.

  Property name choice: Serilog's LogEventProperty constructor accepts

  name and CompactJsonFormatter emits it as a top-level JSON key,
  matching DD's expected field name verbatim.

[2026-05-15] Resilient MCP client reconnect — no more manual restarts
  Previously: McpClient.CreateAsync ran once at startup and the resulting
  client + tool list were captured into the AIAgent singleton's closure.
  When mcp-server-dotnet restarted for ANY reason (rollout, OOM,
  AKS rebalance, image pull retry), the cached client's Mcp-Session-Id
  no longer mapped to anything on the new pod. Every subsequent /query
  hit ClientTransportClosedException → "Error: Function failed" in the
  LLM response. The only mitigation was a manual
  `kubectl rollout restart deployment/agent-api-dotnet`.

  Now: AgentService catches MCP session-expired exceptions, triggers a
  reconnect through a holder, rebuilds the agent with the fresh tool
  list, and retries the request once — transparent to the caller. First
  call after an mcp-server restart pays one extra round trip; every
  call after is normal.

  Files added:
    services/agent-api-dotnet/Services/McpClientHolder.cs
      Owns the McpClient lifecycle + tool list. GetClientAsync /
      GetToolsAsync lazy-connect; RefreshAsync disposes + recreates.
      Exposes a monotonic Generation counter incremented per successful
      (re)connect so callers can cache against it. Thread-safe via
      SemaphoreSlim — concurrent reconnect requests coalesce.

    services/agent-api-dotnet/Services/AgentHolder.cs
      Holds the current ChatClientAgent. ChatOptions.Tools is captured
      at construction so the agent must be rebuilt to pick up a fresh
      tool list after McpClientHolder.RefreshAsync. Caches by
      McpClientHolder.Generation — exactly one rebuild per reconnect.
      Construction happens outside the lock to avoid blocking siblings.

[2026-05-15] Pure-OTel pivot: drop DD .NET tracer, fix MCP, attribute-based DBM
  Three related fixes that together resolve the trace-split bug observed
  during live verification and finalize the .NET stack as 100% OTel-
  instrumented (matching the project's stated AECOM demo intent):

  1. MCP session affinity (commits 3733111 + 5b719db revert):
     ModelContextProtocol.AspNetCore 1.3.0 default is session-stateful;
     each tool/call request carries an Mcp-Session-Id bound to a specific
     server pod. Multi-replica round-robin causes follow-up requests to
     hit the wrong pod → HTTP 400 JsonRpcError → LLM tool callback
     returns "Function failed". Tried Stateless=true on the server
     (commit 5b719db) but the .NET MCP client library still issues
     session-aware requests → 400. Reverted Stateless=true and added
     sessionAffinity:ClientIP (timeoutSeconds:10800) on the K8s Service
     so every agent-api-dotnet request pins to the same server pod for
     3h. Verified live: get_contract_awards now returns real responses;
     no more "Function failed" errors.

  2. DD .NET tracer removal (this commit):
     The DD .NET tracer (admission-injected via dotnet-lib.version:v3)
     ran in parallel with our OTel SDK pipeline and split trace_ids on
     every request — DD tracer read RUM's legacy x-datadog-trace-id
     header (64-bit, padded to 128) while OTel SDK generated its own
     W3C trace_id ignoring that header. invoke_agent and its children
     emitted on the DD-extracted trace_id; classify_domain / retrieval /
     embeddings emitted on the OTel-generated one. Result: trace tree
     literally cut in half across two trace_ids in DD UI. Removing the
     admission annotation eliminates the conflicting tracer entirely —
     OTel SDK reads RUM's W3C traceparent natively → unified trace_id

  downstream exporters see ml_app:

    processors:
      transform/llmobs:
        error_mode: ignore
        trace_statements:
          - context: resource
            statements:
              - set(attributes["ml_app"], attributes["service.name"])
    service:
      pipelines:
        traces:
          processors: [transform/llmobs, batch]
          exporters: [debug, datadog, datadog/connector, otlphttp/llmobs]

  refactor now pays dividends. Pointing OTEL_EXPORTER_OTLP_ENDPOINT at a

  collector's exporters + processor list would change. ml_app becomes
  harmless metadata in any other backend.

[2026-05-12] Delete autoinstr experiment; library auto-instrumentation is what we wanted (af5623b)

  User-initiated terminology clarification resolved several days of
  wrong-direction work. The conflated term "autoinstrumentation"
  actually covers two very different mechanisms:

    Auto-INJECTION (what dotnet-otel-poc-autoinstr/ tried):
      CLR profiler hooks the .NET runtime via CORECLR_PROFILER_* env
      vars, rewrites IL at startup. Requires init container + shared
      volume + arch-matched native .so. Production conflict with
      DD's admission-controller .NET tracer (also a CLR profiler).

    Library auto-INSTRUMENTATION (what we actually wanted):
      NuGet packages (OpenTelemetry.Instrumentation.AspNetCore, .Http,
      …) subscribe to .NET's built-in DiagnosticListener / Activity
      events and emit OTel spans. Plain managed code, no profiler,
      no Docker complexity.

  Modern .NET libraries (ASP.NET Core, HttpClient, EF Core, Azure SDK,
  SqlClient, Npgsql, gRPC, Redis, Kafka, MongoDB) all natively emit
  OTel-compatible events — the instrumentation packages just subscribe.

  Findings from the auto-injection dead-end (useful reference if anyone
  ever needs to go back down this path):

  • platform: linux/amd64 workaround triggers a second issue: QEMU's
    Rosetta emulation can't fully run the .NET 10 SDK's MSBuild
    evaluation inside docker build (MSB4184).
  • Production K8s would still conflict with DD's CLR profiler injection.
